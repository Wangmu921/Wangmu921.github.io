# 操作系统6：多处理器调度

## 🌐 一、 核心背景与多核挑战

随着多核处理器（Multicore Processor）的兴起，多个 CPU 核心被封装在同一个芯片上，多处理器调度成为现代操作系统的标配。
* **软件层面的本质规律**：仅仅增加硬件核心数量并不会让单线程应用程序自动运行得更快。要充分利用多核架构的并行计算红利，必须重写应用程序，将其拆分为多个能够并行运行的**线程（Threads）**。
* **多处理器调度器的核心任务**：如何在错综复杂的硬件拓扑结构（多核、共享/独占缓存、统一内存）之上，合理地为多线程或多进程分配物理 CPU 核心。

---

## 🛑 二、 硬件机制：多核带来的三大物理难题

多处理器调度之所以比单核调度复杂得多，其根本原因在于硬件的**缓存（Cache）层级架构**以及多核同时访问内存时引入的冲突。

### Problem 1：缓存一致性问题 (Cache Coherence)
* **硬件背景**：为了弥补 CPU 飞速计算与主存（Main Memory）慢速访问之间的巨大鸿毛，每个 CPU 核心都配备了小型、高速的硬件缓存（如 L1、L2 Cache），利用**时间局部性**与**空间局部性**存放常用数据的副本。
* **危机场景**：
    1. 物理内存的同一地址（如地址 1）存放着共享数据 $D$。
    2. CPU 0 读取了该数据，将其存入自己的私有缓存中。
    3. 随后，CPU 0 将该值修改为了 $D'$，但该更新**仅仅存在于 CPU 0 的私有缓存中**，尚未写回主存。
    4. 此时，CPU 1 被调度去读取内存地址 1 的数据，由于它无法直接感知 CPU 0 内部缓存的变动，读取到的仍然是主存中的旧数据 $D$。
* **解决方案（总线嗅探 / Bus Snooping）**：
    * **机制**：每个核心的缓存控制器都通过硬件总线（Bus）持续监听（嗅探）其他处理器发出的内存读写请求、读独占或写失效等一致性事务。
    * **动作**：一旦 CPU 0 修改了数据，其他核心监听到这一变化后，会立刻将自己缓存中对应的数据项标记为**失效（Invalidate）**。当 CPU 1 再次需要该数据时，就会被迫触发缓存缺失（Cache Miss），转而从写回后的主存或通过缓存间传输获取最新的正确值 $D'$。

### Problem 2：同步与互斥问题 (Synchronization)
* **危机场景**：当多核缓存一致性问题通过硬件解决后，软件层面的并发冲突接踵而至。如果两个不同的核心（如 CPU 0 和 CPU 1）在同一个瞬间，同时去执行一段无保护的链表删除操作（`List_Pop`）：
    1. 它们可能在微观上同时将指针 `tmp` 指向当前的头节点 `head`。
    2. 它们同时读取了相同的 `head->value`。
    3. 它们又同时将 `head` 推进到 `head->next`。
    4. 最终，相同的内存节点会被 `free()` 释放两次，彻底破坏内存底层的完整性，引发内核崩溃。
* **解决方案（互斥锁 / Mutex）**：
    * 软件级代码必须显式采用**互斥（Mutual Exclusion）**机制。通过在临界区前后加锁（`lock(&m)`）与解锁（`unlock(&m)`），确保在任意一个物理瞬间，只有一个 CPU 核心能够进入该代码段执行，以此守护多核系统下的线程安全。

### Problem 3：缓存亲和性问题 (Cache Affinity)
* **物理规律**：一个进程/线程在某个固定的 CPU 核心上运行一段时间后，该核心的 L1/L2 缓存中就会沉淀下该进程大量的数据和状态信息。
* **性能损耗**：如果调度器在下一个时间片随意切换，将该进程迁移到另一个崭新的 CPU 核心上运行，它在新核心上运行时就会遭遇全场的缓存失效（Cache Miss），必须重新从慢速的主存中搬运数据。而原核心里的缓存数据也会被无情覆盖，造成巨大的性能浪费。
* **设计原则**：多处理器调度器在做出决策时，必须具备**缓存亲和性（Cache Affinity）**意识——**尽可能保持一个进程在相同的 CPU 核心上运行**，减少跨核迁移。

## ♟️ 三、 两大经典调度方法论

为了在平衡负载的同时兼顾可扩展性与亲和性，历史上演进出了两种截然不同的多处理器调度框架：

### 方法 1：单队列多处理器调度 (Single-Queue Multiprocessor Scheduling, SQMS)
* **设计**：最直观的设计。将整个操作系统中所有准备就绪、可运行的任务全部塞进**一个全局共享的单一队列**中。每个物理 CPU 核心一旦空闲，就从这个全局队列中摘取下一个任务下来执行。
* **优势**：实现极其简单。无需复杂的负载均衡策略，天然做到了完美的任务轮流分配。
* **致命短板**：
    1. **缺乏可扩展性（Lock Contention）**：由于整个系统只有一把全局队列锁，当 CPU 核心数增加（如 8 核、16 核、甚至更多）时，所有核心为了拿任务会卡在同一个队列锁上疯狂竞争。系统绝大部分精力都浪费在锁等待上，性能发生剧烈退化。
    2. **亲和性极低**：在没有任何干预的纯净轮转下，任务 $A, B, C, D, E$ 会在不同的时间片随机地被捞到 CPU 0、CPU 1、CPU 2 等各个核心上交替运行，导致缓存亲和性彻底丧失。
    * *改良策略*：可以在 SQMS 中引入启发式亲和性调度，尽量让 $A, B, C, D$ 固定在各自的核心，只有部分特殊任务（如 $E$）充当消防员跨核迁移。但这种打补丁的方式实现起来异常冗余且复杂。

### 方法 2：多队列多处理器调度 (Multi-Queue Multiprocessor Scheduling, MQMS)
* **设计**：彻底走向分布式。系统由**多个独立的调度队列**组成（通常每个 CPU 核心独占一个本地就绪队列，例如 CPU 0 对应 Q0，CPU 1 对应 Q1）。
* **运作机制**：当一个新进程进入系统时，它会通过哈希或特定规则被分配到其中一个本地队列中。此后，每个 CPU 核心只盯着自己的小队列，在内部独立执行轮转（RR）或其他单核调度策略。
* **核心优势**：
    1. **极致的可扩展性**：各个核心的加锁和状态信息完全隔离，消除了全局锁竞争，CPU 核心数越多，吞吐量优势越明显。
    2. **天然的缓存亲和性**：任务（如 $A$ 和 $C$）一旦落入 Q0，就会始终由 CPU 0 宠幸，数据长驻 CPU 0 的本地缓存，命中率极高。

## ⚖️ 四、 MQMS 的终极痛点：负载不均与动态迁移

多队列架构在带来高扩展性的同时，其本质缺陷在于**打破了全局视角的公平性，极易产生负载不均（Load Imbalance）**。

### 1. 负载不均的典型场景
* **常见由于任务消亡**：假设初始状态下，Q0 队列里排着任务 $A$ 和 $C$，Q1 队列里排着 $B$ 和 $D$。大家非常均衡。
* **失衡演进**：
    * 如果排在 Q0 的任务 $C$ 突然运行结束退出了，那么 Q0 队列里就只剩下任务 $A$，而 Q1 里依然有 $B$ 和 $D$。此时，任务 $A$ 将独自霸占 CPU 0 的 100% 资源，而 $B$ 和 $D$ 只能在 CPU 1 上各分到 50% 资源。
    * 更极端的情况下，如果 Q0 里的 $A$ 和 $C$ 全部退出寿终正寝了，**CPU 0 将彻底陷入空闲（Idle）无事可做**，而另一边的 CPU 1 却还在满载超负荷运转。这严重违背了多核设计的初衷。

### 2. 破局之道：动态任务迁移 (Migration)
为了解决上述失衡，操作系统内核必须在统一的框架下，引入跨队列的**负载均衡与任务迁移机制**：

* **基础迁移**：当系统检测到 Q0 为空、Q1 挤压时，操作系统会强制将 Q1 中的任务（如 $B$ 或 $D$）剥离，搬运到 Q0 队列中投喂给 CPU 0，实现微观上的算力救灾。
* **复杂连续交换（多任务混杂）**：如果遇到更复杂的非整除场景（如 CPU 0 只有长任务 $A$，CPU 1 却挤压了 $B$ 和 $D$），单一的一次性迁移无法完美平摊。
    * 调度器会制定出**持续动态交换策略**：在时间轴的前段，让 CPU 0 跑 $A$，CPU 1 跑 $B$ 和 $D$；运行一段时间后，将 $B$ 强行迁移到 CPU 0（使其运行 $A$ 和 $B$），同时将 $A$ 迁移到 CPU 1（使其运行 $A$ 和 $D$）。通过跨核的高频、交替重组，在宏观上逼近完美的系统级负载均衡。

## 🛠️ 五、 课件级 Linux 多处理器硬核实操

通过一系列工业级 Linux 指令，展示了如何在真实系统中洞察和控制多核调度：

### 1. 线程 count 与轻量级进程（LWP）审查
在 Linux 系统中，线程被视为轻量级进程（Light Weight Process, LWP），每个线程在内核中都有唯一的 LWP ID（或称 TID）。
* `ps -o thcount,nlwp \`pidof cpu\``：
    * `thcount` (Thread Count)：该进程当前拥有的总线程数。
    * `nlwp` (Number of Light Weight Processes)：该进程对应的 LWP 数量，两者的数值在常规进程中完全等价（单线程程序均显示为 1）。
* `ps -L -p \`pidof threads-s\``：
    * `-L` 参数能将该进程内部的所有隐秘线程全部平铺打印出来。你能清晰地看到即使它们的 `PID`（进程ID）完全相同，但每个人都拥有独特的、作为线程唯一标识的 `LWP` 编号。

### 2. 查看线程究竟在哪个物理核心上跳舞
* `ps -mo pid,tid,%cpu,psr \`pidof threads-s\``：
    * 这是一个极其硬核的多核观测命令。
    * `tid`：区分同一进程内部不同线程的线程 ID。
    * `psr` (Processor)：**该线程在当前这个瞬间，正躺在系统的哪一个物理 CPU 核心（如 Core 0、Core 1）上被执行。**
* `watch -n 0.5 "ps -mo pid,tid,%cpu,psr \`pidof threads-s\`"`：每隔 0.5 秒高频刷新，你就能亲眼目睹随着调度器负载均衡的介入，某个线程的 `PSR` 核心编号是如何在多核之间跳跃闪烁的。

### 3. 强行锁定核心：多核环境下的亲和性强加
如果你在代码中写了多线程，但想做控制变量实验，或者想在工业级调优中彻底避免跨核迁移带来的缓存内伤，可以使用 `taskset` 工具：
* `taskset -c 1 ./threads-s 10000000`：
    * `-c 1`：**强行指定 1 号 CPU 核心。**
    * 作用：操作系统内核的调度器将受到硬性行政干预，该命令启动的所有子线程 p1、p2 以及主线程，将被**死死锁死在 1 号 CPU 核心上**，绝对不允许跨核迁移到 0 号或 2 号核心。此时，这个多线程程序将在 1 号核心上退化为纯单核分时轮转执行。
