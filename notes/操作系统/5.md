# 操作系统5：进程调度 - 多级反馈队列与比例份额

## 🗺️ 一、 多级反馈队列 (MLFQ) 核心思想

多级反馈队列（Multi-Level Feedback Queue, MLFQ）是为了解决通用操作系统中**无法预知进程运行时间**的现实矛盾而设计的经典调度算法。

* **设计目标**：
    1.  **改善周转时间**：通过优先服务短作业或表现出交互型特征的进程，近似实现 SJF/STCF 的效果。
    2.  **降低响应时间**：在对任务长度一无所知的前提下，尽量让新任务和交互式任务快速获得 CPU 响应。
* **反馈（Feedback）的本质**：MLFQ 根据进程**过去的 CPU 使用行为**动态调整其优先级，以此来“预测”其未来的行为模式。

---

## 📜 二、 MLFQ 基本规则与演进过程

### 1. 基础规则（未考虑作弊与饥饿）
MLFQ 由多个具有不同优先级（Priority Level）的就绪队列组成，进程在任一时刻属于其中一个队列。
* **规则 1**：若 $\text{Priority}(A) > \text{Priority}(B)$，则进程 $A$ 运行，进程 $B$ 不运行。
* **规则 2**：若 $\text{Priority}(A) == \text{Priority}(B)$，则进程 $A$ 和 $B$ 以轮转法（Round Robin, RR）机制交替运行。
* **规则 3**：当一个新进程进入系统时，它被强制放置在**最高优先级队列**中。
* **规则 4a**：如果一个进程在运行时**用完了整个时间片**，说明它属于 CPU 密集型任务，其优先级将被**降低一级**（下移到较低队列）。
* **规则 4b**：如果一个进程在时间片结束之前**主动放弃了 CPU**（例如因为发起 I/O 操作），它将**保持在当前的优先级**水平。

---

### 2. 传统 MLFQ 的三大致命漏洞与防御升级

传统的基础 MLFQ 规则在错综复杂的真实业务场景中，会遭遇以下严重问题：

#### 漏洞 A：进程饥饿问题 (Starvation)
* **现象**：如果系统中有“太多”短时运行的交互式进程不断涌入，高优先级队列将永远不为空，导致躺在低优先级队列中的长周期 CPU 密集型进程**永远分配不到 CPU 时间**。
* **升级方案（规则 5 - 优先级提升 / Priority Boost）**：
    * **策略**：每经过一段固定的时间 $S$，调度器强行将系统中**所有的进程全部移动到最高优先级队列**中。
    * **效果**：完美解决了长任务饿死的问题；同时，如果一个老进程从“CPU 密集型”转变成了“I/O 密集型”，也能借此机会重返高优先级舞台。

#### 漏洞 B：利用调度规则作弊 (Gaming the Scheduler)
* **现象**：恶意程序可以在一个时间片快要结束前的临界点（比如消耗了 99% 的时间片时），故意发起一次无意义的 I/O 操作（如读写空设备）来释放 CPU。
* **后果**：由于它在时间片结束前放弃了 CPU，触发了原版的**规则 4b**，导致其优先级得以保留在高级队列中。通过这种反复欺骗，该进程将恶意霸占极高比例的 CPU 资源。
* **升级方案（重写规则 4 - 记录每一级的累积 CPU 时间）**：
    * **策略**：取消原来的 4a 和 4b，将其合并重写为：**一旦一个进程在某个优先级队列中消耗完了规定的累积时间片总量（无论其中途主动放弃了多少次 CPU），其优先级就必须降低一级。**
    * **效果**：杜绝了靠高频 I/O 刷优先级的作弊手段，确保了 CPU 资源分配的公平性。

---

### 3. MLFQ 的工程参数配比艺术
在真实的操作系统（如 Solaris、FreeBSD）中，MLFQ 的时间片并不是一成不变的，通常采用**可变时间片**的设计：
* **高优先级队列**：分配**较短的时间片**（例如 10ms 或更少）。因为交互式任务需要的是快速响应，单次计算量小，短时间片可以保证多个交互任务频繁切换。
* **低优先级队列**：分配**较长的时间片**（例如 40ms 或更多）。因为沉淀到低优先级的都是长周期的计算密集型任务，给它们分配长一点的时间片，可以有效减少上下文切换造成的硬件开销（如破坏 Cache 局部性和 TLB 缓存状态）。

## 🎟️ 三、 比例份额调度 (Proportional Share)

比例份额调度器（又称公平共享调度器 / Fair-Share Scheduler）关注的核心不是优化周转时间或响应时间，而是**保证每个进程能获得期望中特定百分比的 CPU 运行时间**。

### 方法 1：彩票调度 (Lottery Scheduling)
* **彩票（Ticket）的含义**：代表进程或用户应获得的资源份额。进程拥有的彩票数占系统总票数的百分比，就是它理论上能分到的 CPU 时间比例。
    * *示例*：进程 $A$ 持有 75 张票（编号 0~74），进程 $B$ 持有 25 张票（编号 75~99）。
* **调度机制**：每个时间片结束时，调度器利用随机数发生器抽取一张中奖彩票（例如产生 0~99 之间的随机数），抽到谁的号码谁就上 CPU 运行。
* **随机性的优势**：
    1.  **无状态边界情况**：避免了传统确定性算法在特定临界状态下的反常行为。
    2.  **极端轻量**：调度器不需要为每个进程维护复杂的全局运行状态，只需要在链表遍历中累加票数匹配中奖号码。
    3.  **大数定律支持**：任务运行时间越长，它们实际得到的 CPU 时间比例就越趋近于理想的期望比例。

#### 🎫 彩票调度的特殊高级机制
1.  **彩票货币 (Ticket Currency)**：允许用户在多用户系统中，在自己的本地使用任意面额的“本地货币”分配给子进程。系统内核会自动根据该用户拥有的“全局彩票”权重，将其等比例换算为正确的全局统一彩票。
2.  **彩票转让 (Ticket Transfer)**：一个进程可以暂时把自己的彩票全部交托给另一个进程。这在“客户端-服务器（Client-Server）”架构中极其有用，当服务器代表客户端搬砖时，能临时继承客户端的优先级权重。
3.  **彩票通胀 (Ticket Inflation)**：进程在相互信任的信任链条里，可以自行临时增加或减少自己持有的彩票总数，用以应对突发的计算洪峰。

---

### 方法 2：步长调度 (Stride Scheduling)
彩票调度最大的痛点在于**随机性在短周期任务内极不精准**（可能连续好几次中奖号码都偏向某一个进程）。为了在**每一个调度周期**后都做到完全精确的绝对公平，Waldspurger 发明了确定性的步长调度。

* **核心物理量计算**：
    1.  **步长（Stride）**：
        $$\text{Stride} = \frac{\text{一个非常大的系统常数（如 10000）}}{\text{该进程持有的票数}}$$
        *票数越多，步长越小，代表其前进所需的阻力越小。*
    2.  **行程（Pass）**：每个进程内部维护的一个全局计数器。
* **调度策略**：
    * 调度器永远挑选当前就绪队列中 **$\text{Pass}$ 值最小**的进程上台运行。
    * 该进程运行完一个时间片后，它的行程值进行累加更新：
        $$\text{Pass}_{\text{new}} = \text{Pass}_{\text{old}} + \text{Stride}$$
    * 更新后将进程放回就绪队列，周而复习。
* **步长算法的致命破绽**：当一个**新进程**突然加入系统时，它的初始 $\text{Pass}$ 值通常被设为 0。这会导致新进程在接下来的很长一段时间内，由于 $\text{Pass}$ 值严重偏低而**死死垄断住 CPU 资源**，将全场老进程全部打入冷宫。这是它无法取代通用调度器的关键原因。

---

## 🚀 四、 工业级终极落地：Linux 完全公平调度器 (CFS)

Linux 系统的默认调度器 **CFS (Completely Fair Scheduler)** 彻底抛弃了传统的时间片和优先级队列，完全基于**比例份额**的思想来实现。

### 1. 核心武器：虚拟运行时间 (`vruntime`)
* **概念**：每个进程在 PCB（在 Linux 中为 `task_struct`）中都有一个名为 `vruntime` 的流逝计数器。
* **选人规则**：当需要发生调度切换时，CFS 永远挑就绪队列中 **`vruntime` 最小**的那个进程上台。
* **更新机制**：进程在物理 CPU 上实际运行了一段真实时间 $\text{runtime}$ 后，它的虚拟运行时间按如下公式增长：
    $$\text{vruntime}_{\text{new}} = \text{vruntime}_{\text{old}} + \text{runtime} \times \left( \frac{\text{Weight}_0}{\text{Weight}_i} \right)$$
    *其中 $\text{Weight}_0$ 是 nice 值为 0 时的基准权重（1024）。*

### 2. 权重掌控者：Nice 值映射表
Linux 沿用了 Unix 的 Nice 值机制，范围为 $[-20, +19]$。**Nice 值越低（负数），优先级越高，在内核映射表中的权重值就暴增。**
* *示例*：
    * $\text{Nice} = -5 \rightarrow \text{权重} = 3121$
    * $\text{Nice} = 0 \rightarrow \text{权重} = 1024$
* 当这两个进程共同平分系统时，$\text{Nice}=-5$ 的进程由于权重非常大，其分母优势明显，每次运行后 `vruntime` 的增长速度只有普通进程的约三分之一。所以它的 `vruntime` 总是全场最小，从而能获得大得多的 CPU 物理执行份额。

### 3. CFS 核心控制参数
* `sched_latency`（调度延迟）：一个调度周期的时间总量（如 48ms）。CFS 会保证在这段时间内，全场 $n$ 个进程必须轮流上台跑完一遍。
    $$\text{每个进程分到的物理时间片} = \text{sched\_latency} \times \left( \frac{\text{Weight}_k}{\sum \text{Weight}_i} \right)$$
* `min_granularity`（最小粒度限制）：防止进程过多时（如 $n$ 极大），时间片被无限稀释（例如变成 0.1ms），导致系统全在做上下文切换。CFS 规定任何进程单次运行的物理时间绝对不能低于该下限（通常设为 6ms）。

### 4. 解决长眠进程被唤醒后的“CPU 暴击”问题
* **潜在危机**：如果一个进程长时间睡眠或做 I/O，它的 `vruntime` 会停滞不前。一旦它突然被唤醒，其 `vruntime` 将远低于全场一直在辛勤劳作的进程。如果不加控制，它将长期霸占 CPU，造成严重的**系统卡顿**。
* **CFS 的解决策略**：当睡眠进程被唤醒重新加入调度树时，内核会强制将它的 `vruntime` 强行拉高重置，将其设置为当前**红黑树中所有就绪进程的最小 `vruntime` 值**。这样既能保证它被立刻调度，又防止了它对 CPU 进行恶意的补偿性垄断。

### 5. 高效底层实现：红黑树 (Red-Black Tree)
* 为了能在长达几千个进程的就绪队列中，以极致的性能找出 `vruntime` 最小的进程，CFS 放弃了传统的线性链表，在内核中维护了一棵**以 `vruntime` 作为 Key 的自平衡二叉树——红黑树**。
* **时间复杂度**：
    * 寻找下一个运行进程：只需要直接摘取红黑树最左侧的叶子节点，时间复杂度为 $O(1)$（通过缓存指针）。
    * 新进程插入或旧进程重新入队：树的自平衡调整时间复杂度控制在对数级 $O(\log n)$。

## 🛠️ 五、 课件中提及的系统级硬核实操命令

演示如何通过 Linux 命令行工具直接操纵和观察内核的调度器：

1.  **查询和调整正在运行任务的 Nice 值 (SCHED_OTHER策略)**
    * `sudo nice --10 ./cpu hello`：以高特权级、Nice 值为 -10（高优先级）的身份启动 `cpu` 进程。
    * `sudo renice -n 0 -p <PID>`：将正在后台运行的指定 PID 进程的 Nice 值动态重置回 0。
    * 通过 `htop` 命令可以直接在 `NI` 列看到动态修改的结果。
2.  **强制切换为硬核实时调度策略 (Real-time Policies)**
    * `chrt -p <PID>`：查询该进程当前采用的调度策略。通用程序通常显示为常规的公平分时策略 `SCHED_OTHER`。
    * `sudo chrt -r -p 10 <PID>`：直接剥夺该进程在 CFS 下的公平份额，将其**强行提升为硬核实时轮转策略 (SCHED_RR)**，并将实时静态优先级设为 10（实时范围为 1~99）。此时该进程将彻底凌驾于所有常规 CFS 进程之上，由内核实时守护。
3.  **窥探内核调度器的秘密账本**
    * `cat /proc/sys/kernel/sched_rr_timeslice_ms`：查看系统默认的实时调度时间片长度。
    * `cat /proc/<PID>/sched`：动态观察并打印出当前进程的全部调度细节。能直接看到该进程的虚拟运行时间 `se.vruntime`、总执行物理时间 `se.sum_exec_runtime`、以及由于时间片用完被内核强行踢下台的非自愿上下文切换次数 `nr_involuntary_switches`。
