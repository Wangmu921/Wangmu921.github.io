# 📝 操作系统3：受限直接执行机制（LDE）

## 一、 引言：控制与性能的博弈

### 1. 如何高效地控制虚拟化 CPU？
操作系统为了让成百上千个进程在有限的 CPU 上运行，采用了**时分共享（Time sharing）**的方式：运行一个进程一段时间，然后运行另一个进程，如此轮换 。但在实现虚拟化时，操作系统面临两个核心挑战 ：
* **性能（Performance）：** 如何在不增加系统开销的情况下实现虚拟化 ？如果操作系统的每一步都要通过软件模拟来介入，性能会极低 。
* **控制（Control）：** 如何有效地运行进程，同时保留操作系统对 CPU 的绝对控制权 ？防止进程独占 CPU 或破坏系统 。

为了完美解决这两个问题，操作系统开发人员与硬件厂商联手想出了一种底层技术：**受限直接执行（Limited Direct Execution, LDE）** 。

### 2. 基础技术：直接执行（Direct Execution）
如果没有限制，最直观的“直接执行”协议非常简单：**只需直接在 CPU 上运行程序即可** 。

| 操作系统（内核模式） | 程序（用户模式） |
| :--- | :--- |
| 1. 在进程列表上创建条目 <br>2. 为程序分配内存并加载到内存中 <br>3. 根据 `argc`/`argv` 设置程序栈 <br>4. 清除寄存器 <br>5. 执行 `call main()` 方法  | |
| | 6. **执行 `main()` 方法** <br>7. **从 `main` 中执行 `return`**  |
| 8. 释放进程的内存 <br>9. 将进程从进程列表中清除  | |

> ⚠️ **无限制执行的致命缺陷：**
> 如果不对运行的程序进行任何限制，操作系统将失去对任何事情的控制权，从而退化为**“仅仅是一个函数库（just a library）”** 。此时程序可以随意修改任意内存、永久独占 CPU 或破坏其他进程 。

## 二、 问题 1：如何执行受限操作（特权划分与系统调用）

如果进程希望执行某种受限操作（特权操作），例如：向磁盘发出 I/O 请求 、获得更多系统资源（如 CPU 或内存） ，如果让进程做任意它想做的事情，OS 将无法构建文件安全、保护等期望的功能 。

### 1. 方案：双重执行模式（Dual-mode Execution）
硬件通过提供两种不同的执行模式来协助操作系统（通常对应 CPU 的**特权级别/特权环 Ring**） ：
* **用户模式（User mode / Ring 3）：** 应用程序运行在此模式下，**不能完全访问硬件资源** 。如果尝试执行特权指令，会触发硬件异常 。
* **内核模式（Kernel mode / Ring 0）：** 操作系统运行在此模式下，**可以访问机器的全部资源** 。

### 2. 核心机制：系统调用（System Call）与 Trap 机制
用户进程如何使用特权操作？答案是：**用户程序执行系统调用，由内核代理执行操作** 。
系统调用允许内核小心地向用户程序暴露某些关键功能，包括：访问文件系统、创建/销毁进程（如 `fork()`）、进程间通信（如 `pipe()`）、分配更多内存（如 `malloc()`）等 。
为了实现从用户态到内核态的**受保护的控制权转移**，硬件提供了两条关键指令 ：

#### ① Trap 指令（陷入内核）
* **动作：** 跳入内核，同时**将特权级别提升到内核模式** 。
* **硬件自动保存上下文：** 执行 Trap 指令时，必须确保足够的信息被硬件存储，以便准确返回 。CPU 会**自动保存关键上下文信息**（如程序计数器 PC、标志寄存器、其他重要寄存器）到进程的**内核栈（Kernel stack）**中 。
* **硬件架构示例：** x86 中的 `INT` 指令，x86-64 中的 `SYSCALL` 指令，AArch64 中的 `SVC 0` 指令 。

#### ② Return-from-trap 指令（陷阱返回）
* **动作：** **从内核栈弹出（pop）**之前保存的寄存器值，恢复用户模式程序的执行 。
* **降权：** **降低特权级别到用户模式**，并跳转到发起调用的用户程序（Trap 之后的下一条指令位置） 。
* **硬件架构示例：** x86 中的 `IRET` 指令，x86-64 中的 `SYSRET` 指令 。

## 三、 关键底层数据结构：陷阱表（Trap Table）

用户程序在触发 `trap` 时，不能允许它指定跳转到内核的任意位置（否则程序可以跳过内核的安全检查）。为了保证安全性，必须采用**统一的入口机制**。

### 1. 陷阱表的作用
* **存储各类异常、中断和系统调用的处理程序（Handler）入口** 。
* **保证用户程序不能直接执行内核代码：** 用户程序只能给出一个中断向量号/系统调用号，由 CPU 查找该表并跳转 。
* **实现分类：**
    * 在 x86 实模式（Real Mode）下，被称为 **IVT（Interrupt Vector Table，中断向量表）**，位于内存低地址（`000000H` 到 `0003FFH` 共 1024 字节），包含 256 个表项，每项 4 字节 。
    * 在 x86 保护模式/64位下，被称为 **IDT（Interrupt Descriptor Table，中断描述符表）** 。

### 2. 硬件中的事件处理流程
当事件（除零错误、硬件中断如键盘/时钟、系统调用）发生时 ：
$$\text{事件发生} \longrightarrow \text{CPU 获得中断向量号 (Vector number)} \longrightarrow \text{查找 Trap Table / IDT} \longrightarrow \text{跳转到对应 Handler} \longrightarrow \text{执行内核代码} \quad \text{}$$

#### 附：硬件中的中断/异常分类（x86 体系） 
* **中断（Interrupt）：** **异步**硬件事件。例如 I/O 操作完成、显式的 `INT` 指令调用 。
* **异常（Exception）：** 当错误条件发生时的**同步**事件，分为三种 ：
    * *Traps（陷阱）：* 立即报告，处理完后**继续执行下一条指令**（如系统调用） 。
    * *Faults（故障）：* 立即报告，处理完后**重新执行触发故障的指令**（如缺页异常 Page Fault） 。
    * *Abort（中止）：* 严重错误发生，硬件部分失败（如蓝屏死机），程序直接终止 。

## 四、 问题 2：进程间切换（重新获得 CPU 控制权）

> 🤔 **核心矛盾：**
> 当一个用户进程正在 CPU 上运行时，意味着 CPU 当前**没有在执行内核代码**（假设单核） 。那么，操作系统如何重新获得 CPU 的控制权，以便在进程之间进行切换呢 ？

操作系统开发历史中演进出了两种主要方法 ：

### 1. 合作式（Cooperative）方法：等待系统调用
* **核心思想：** 相信应用程序是善良的，**等待系统调用主动让出控制权** 。
* **转交时机：**
    * 进程通过执行系统调用（如 `yield()`）周期性地自愿让出 CPU 。
    * 应用程序在执行**非法操作**（如除零错误、尝试访问不应访问的内存）时，会被硬件捕获并强制将控制权转交给 OS 。
* **历史代表：** 早期版本的 Macintosh 操作系统、老版 Xerox Alto 系统 。
* **致命缺陷：** **如果进程陷入无限循环（死循环），系统将彻底卡死！** 唯一的解决方法是**重启计算机** 。

### 2. 非合作式（Non-Cooperative）方法：时钟中断（强制抢占）
* **核心思想：** 操作系统引入硬件定时器，**强行接管控制权** 。
* **时钟中断（Timer interrupt）机制：**
    * 硬件组件（在 x86 中称为可编程中断时钟 PIT）每隔一定的毫秒数就会**强行触发一个硬件中断** 。
    * **系统启动时：** 内核为定时器中断设置中断处理程序（Timer handler），并启动时钟 。
    * **时钟中断触发时：** 当前运行的进程**被强制抢占**，硬件自动保存被暂停进程的状态，并跳转运行内核中的时钟中断处理程序 。
* **意义：** 定时器中断确保了操作系统能够**定期、强制地获得 CPU 控制权**，从而彻底解决了死循环霸占 CPU 的问题 。

## 五、 进程切换的终极演级：上下文切换（Context Switch）

当时钟中断使操作系统重获控制权后，内核的**调度器（Scheduler）**会根据调度策略（如 FCFS, SJF, STCF, RR 等）决定：是继续运行当前进程，还是切换到另一个进程 。
如果决定切换，操作系统将执行**上下文切换（Context Switch）** 。

### 1. 上下文切换的五大核心步骤
1. **保存当前进程（A）的上下文：** 记录程序计数器（PC）、通用寄存器、标志寄存器以及栈指针（SP） 。
2. **将状态保存至 PCB：** 将上述寄存器现场保存到进程 A 的**进程控制块（PCB / 进程结构体）**中 。
3. **调度器选择新进程（B）** 。
4. **恢复新进程（B）的上下文：** 从进程 B 的 PCB 中将状态、寄存器值、栈指针、程序计数器恢复到 CPU 寄存器中 。
5. **切换地址空间：** 刷新内存映射，执行新进程 B 。

### 2. 深入底层：xv6 上下文切换汇编分析
在著名的 xv6 教学操作系统中，上下文切换的核心函数为 `swtch`，其底层汇编实现（x86 32位架构）如下 ：

```assembly
.globl swtch
swtch:
    # ---------------------------------------------
    # 阶段 1: 保存当前旧进程(Old)的寄存器上下文
    # ---------------------------------------------
    movl 4(%esp), %eax    # 将指向旧进程context指针的指针(struct context **old)加载到eax 
    popl 0(%eax)          # 弹出当前的返回地址(即旧进程的指令指针 IP/PC)，保存到 old context 中 
    movl %esp, 4(%eax)    # 将当前旧进程的栈指针(%esp)保存到 old context 偏移4字节处 
    movl %ebx, 8(%eax)    # 保存 %ebx 寄存器 
    movl %ecx, 12(%eax)   # 保存 %ecx 寄存器
    movl %edx, 16(%eax)   # 保存 %edx 寄存器
    movl %esi, 20(%eax)   # 保存 %esi 寄存器
    movl %edi, 24(%eax)   # 保存 %edi 寄存器
    movl %ebp, 28(%eax)   # 保存 %ebp 栈基址寄存器

    # ---------------------------------------------
    # 阶段 2: 加载新进程(New)的寄存器上下文
    # ---------------------------------------------
    movl 4(%esp), %eax    # 将指向新进程context的指针加载到 eax 
    movl 28(%eax), %ebp   # 恢复新进程的 %ebp
    movl 24(%eax), %edi   # 恢复新进程的 %edi
    movl 20(%eax), %esi   # 恢复新进程的 %esi
    movl 16(%eax), %edx   # 恢复新进程的 %edx
    movl 12(%eax), %ecx   # 恢复新进程的 %ecx
    movl 8(%eax), %ebx    # 恢复新进程的 %ebx 
    
    movl 4(%eax), %esp    # 【关键切栈】将栈指针 %esp 切换为新进程的栈指针！ 
    pushl 0(%eax)         # 将新进程原先保存的返回地址压入新栈中 
    
    ret                   # 执行返回指令，弹出刚才压入的地址并跳转，自此完全进入新进程执行
