# 操作系统8：内存操作 API

## 一、 内存分配的核心分类：用户级库函数 vs 内核级系统调用

在 C 语言程序中，内存分配的接口存在清晰的层级关系。程序员常用的分配接口大多运行在用户态，而底层则由操作系统内核的系统调用提供物理支撑。

### 1. 用户级内存分配接口
* **核心接口**：`malloc()`、`free()`、`calloc()`、`realloc()`。
* **本质**：它们**不是系统调用**，而是标准 C 库（如 Linux 上的 `glibc`）在用户态维护的动态内存分配器库函数。
* **运行机制**：用户级分配器内部维护着一个内存池。当程序调用 `malloc()` 时，分配器首先尝试在自己的动态空闲链表中寻找合适的块分配给程序。

### 2. 内核级内存分配系统调用
当用户级分配器发现其手中的物理内存页或动态空闲块全部耗尽，无法满足程序的动态请求时，它会向操作系统内核发起底层的系统调用，向内核“要地”，从而扩容进程的虚拟地址空间：
* **`brk` / `sbrk` 系统调用**：通过向上或向下移动进程堆荒地的边界指针（Break 指针），直接增加或减少堆空间的大小。
* **`mmap` 系统调用**：向内核请求一块全新的、匿名的虚拟内存映射区域。在 Linux 系统中，当程序请求分配极大的内存块（如大宗数据，通常大于 128KB）时，`glibc` 的 `malloc` 底层会自动放弃 `brk`，转而使用 `mmap` 系统调用来获取独立的虚拟内存区域。

## 二、 核心内存分配 API 详解

### 1. `malloc()` —— 基础堆空间请求
* **函数原型**：`void *malloc(size_t size);`
* **主要职责**：在堆（Heap）上请求分配一块连续的、大小为 `size` 字节的虚拟内存区域。
* **核心特性**：
    * **返回值**：成功时返回指向该内存块起始地址的 `void*` 类型的指针。由于返回的是通用指针，在 C 语言中通常需要将其强制转换为目标数据类型的指针（如 `int *ptr = (int *)malloc(sizeof(int));`）。
    * **失败处理**：如果系统物理内存极度匮乏，导致地址空间无法扩容，`malloc` 会宣告失败并返回 `NULL`。因此，编写硬核健壮的代码时，必须对返回值进行有效性检查（如 `if (ptr == NULL)`）。
    * **无初始化保证**：`malloc` 极其追求速度，**它在分配成功后，绝对不会对这块内存进行任何物理初始化**。这意味着该内存块里保留的完全是之前其他进程消亡前残留的、脏乱的垃圾二进制数据。如果在未赋值前直接读取，会导致程序行为不可预测。

### 2. `calloc()` —— 安全清零分配
* **函数原型**：`void *calloc(size_t num, size_t size);`
* **主要职责**：在堆上分配 `num` 个块、每个块大小为 `size` 字节的连续内存空间（总分配字节数为 `num * size`）。
* **核心特性**：
    * **自动初始化**：与 `malloc` 不同，`calloc` 在成功申请到空间后，**会自动将这块内存中的每一个二进制比特位全部物理清零（初始化为 0）**。
    * **适用场景**：非常适合为数组、矩阵、或需要默认初值清零的结构体动态开辟空间。因为需要执行清零操作，其微观性能开销略高于 `malloc`。

### 3. `realloc()` —— 动态重定位与在线扩容
* **函数原型**：`void *realloc(void *ptr, size_t size);`
* **主要职责**：更改由之前 `malloc`、`calloc` 或 `realloc` 分配的动态内存块的大小。
* **内部底层行为**：
    * **原地扩容**：分配器首先探查当前指针 `ptr` 所指块的正后方，如果紧挨着的堆空间是空闲的且大小足够，则直接原地扩展该块的大小，并返回原指针 `ptr`。
    * **异地搬迁**：如果后方空闲空间不足，分配器会在堆的其他地方重新开辟一块全新的、大小为 `size` 的内存块，将旧内存里的数据原封不动地复制（Copy）到新块中，然后**自动隐式释放掉旧的内存块**，最后返回新内存块的起始地址指针。
    * **失败保持**：若扩容失败，返回 `NULL`，但此时**原始指针 `ptr` 所指向的旧内存和数据依然完好无损，不会被破坏或释放**。

## 三、 常见内存错误与硬核调试隐患

由于 C 语言和操作系统赋予了程序员直接操作物理/虚拟指针的绝对自由，一旦逻辑出现漏洞，就会引发各种灾难性的内存错误。在操作系统课程实验中，这些错误会导致系统频繁崩溃：

### 1. 忘记分配内存即直接使用（野指针与段错误）
* **场景**：定义了一个指针，却未调用 `malloc` 为其分配空间，或者分配失败后未校验返回指针，便直接进行数据写入。
* **代码示例**：
    ```c
    char *src; // 野指针，指向一个完全随机的虚拟地址
    strcpy(src, "hello"); // 灾难！直接踩踏未定义空间，立刻触发系统的 Segmentation Fault 崩溃
    ```

### 2. 分配空间不足导致缓冲区溢出（Buffer Overflow）
* **场景**：分配堆空间时计算错误，导致实际写入的数据长度超出了申请的字节数。
* **最经典隐患（忘记字符串结束符）**：在为字符串动态分配空间时，**忘记给末尾的结束标记符 `\0` 留出 1 字节的空间**。
* **代码示例**：
    ```c
    char *buf = (char *)malloc(strlen(str)); // 错误！strlen 不计算末尾的 '\0'
    strcpy(buf, str); // 写入时，'\0' 会强行溢出到该内存块的外侧，踩坏相邻的动态元数据
    ```

### 3. 忘记初始化内存（Uninitialized Read）
* **场景**：使用 `malloc` 分配堆空间后，在还没有对其进行显式赋值或填装数据的情况下，代码逻辑便直接读取该区域。
* **后果**：程序会直接读到系统残留的二进制垃圾数据，导致后续的条件判断或数学计算发生不可预知的诡异偏转。

### 4. 动态内存泄漏（Memory Leak）
* **场景**：在函数内部或循环体中不断通过 `malloc` 申请堆空间，但程序在失去对该指针的控制前（如函数返回、指针重定向），**从未调用 `free()` 将其归还系统**。
* **后果**：对于长期在后台运行的守护进程或服务器程序，内存泄漏会导致堆空间持续膨胀，最终物理内存和交换空间被彻底耗尽，触发 Linux 操作系统的 **OOM Killer（内存溢出杀手）** 机制，进程会被内核强行强制杀死。

### 5. 双重释放（Double Free）
* **场景**：对同一个动态内存块的指针，在没有重新分配的情况下，连续调用了两次或多次 `free(ptr);`。
* **内部内伤**：用户级分配器（如 `ptmalloc`）在接收到 `free` 请求时，会把该块挂回空闲链表。如果双重释放，会导致分配器内部的空闲链表元数据（如前驱/后继指针）形成恐怖的**环形死循环**，使整个动态内存系统彻底瘫痪，并且极易被黑客用于执行任意代码注入攻击。

### 6. 释放后使用（Use After Free, UAF）
* **场景**：已经显式调用了 `free(ptr);` 将内存归还给了分配器，但由于没有及时将该指针置为 `NULL`，在随后的代码逻辑中，程序依然通过旧指针 `ptr` 尝试去读取或写入数据。
* **后果**：此时这块内存可能已经被重新分配给了系统中的其他关键对象，通过旧指针进行写入会直接踩烂新对象的数据。这在网络安全中是价值极高的白帽高危漏洞。
