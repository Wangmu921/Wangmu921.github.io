# 操作系统4：进程调度 - 介绍

## 🏗️ 一、 核心背景：操作系统内核与受限直接执行

1. **操作系统的构成**：内核（Kernel） + 其他组件 。
2. **内核特权支持**：内核代码运行在**内核态**，这需要底层硬件提供特权级（Privilege Level）的支持。
3. **系统的启动流程**：
   * 计算机开机 $\rightarrow$ 引导加载程序（如 GRUB）加载内核。
   * 内核完成自身的初始化 $\rightarrow$ 启动第一个用户进程（如 `init` 进程） 。
4. **受限直接执行（Limited Direct Execution）**：用户进程平时是以极快的速度直接在物理 CPU 上跑的（即“直接执行”） 。但是它的权限受限，一旦发生**系统调用、硬件中断或异常**，CPU 会立刻强制切回**内核态**，把控制权交还给内核。
5. **调度器（Scheduler）的职责**：操作系统通过调度和上下文切换实现 CPU 共享 。而**调度器就是内核里的“交警”**，它通过特定的策略（Policy），决定在就绪队列中哪一个进程/线程有资格获得接下来的 CPU 时间片。

## 📏 二、 评估调度的两把尺子（指标体系）

要评价一个调度算法是好是坏，需要定义核心的数学指标。本章主要关注以下两个指标：

### 1. 周转时间 (Turnaround Time, $T_{\text{turnaround}}$)
* **定义**：进程从**提交（到达）**系统到它**执行完成**所花费的时间。
* **数学公式**：
  $$T_{\text{turnaround}} = T_{\text{completion}}（完成时间） - T_{\text{arrival}}（到达时间）$$
* **性质**：这是一个**全局指标**，反映了系统处理任务的整体速度。

### 2. 响应时间 (Response Time, $T_{\text{response}}$)
* **定义**：进程从**提交（到达）**系统到它**第一次获得 CPU 并开始运行**所经历的时间。
* **数学公式**：
  $$T_{\text{response}} = T_{\text{first run}}（首次运行时间） - T_{\text{arrival}}（到达时间）$$
* **性质**：这是一个**人机交互指标**。对于现代有界面的交互式系统，响应时间越短，用户感觉系统越流畅。

## ♟️ 三、 调度算法的演进：从理想走向现实

为了逐步推演算法，建立以下 **5 个初始的“完美假设”**：
1. 每一个工作（Job）运行的时间都相同。
2. 所有工作在同一时间到达系统。
3. 一旦工作开始运行，它就会一直运行到结束（不可抢占）。
4. [cite_start]所有工作只使用 CPU，不执行任何 I/O 操作。
5. 每个工作的运行时间（工作量）是系统提前预知（已知）的。

接下来的逻辑就是：**打破假设，升级算法。**

### 1. 先来先服务 (First-Come, First-Served, FCFS)
* **基于假设**：放宽假设 1（工作运行时间不再相同）。
* [cite_start]**调度策略**：最简单的队列逻辑。谁先来，谁就先上 CPU 运行，直到它自己运行完，下一个再上。
* **致命痛点**：**护卫效应（Convoy Effect）**。如果一个运行时间极长的“大卡车”任务稍微排在前面，后面跟着几个运行极快的“小轿车”任务。导致后面的小任务必须死等大任务完工，全队的**平均周转时间（Average Turnaround Time, ATT）**会被疯狂拉长。

### 2. 最短任务优先 (Shortest Job First, SJF)
* **解决策略**：为了解决护卫效应，SJF 规定：**当有多个任务同时在队列中时，优先选择运行时间最短的任务去执行。**
* **打破假设**：放宽假设 2（任务不再是同时到达，而是**陆续到达**）。
* **遭遇困境**：如果大任务 A 在 $t=0$ 时刻首先到达并开始运行，两个小任务 B 和 C 在稍后时刻到达。因为 SJF 此时是**非抢占式（Non-preemptive）**的，CPU 一旦给了 A 就无法收回，A 必须跑完。结果 B 和 C 依然要经历漫长的等待，SJF 算法在非抢占场景下溃败。

### 3. 最短剩余时间优先 (Shortest Time-to-Completion First, STCF)
* **机制升级**：将 SJF 升级为**抢占式（Preemptive）**。
* **调度策略**：每当有新任务到达系统时，调度器会紧急中断当前运行的任务，并重新盘点：**对比全场所有任务“剩余还需要运行的时间”，谁剩下的时间最短，CPU 就给谁！**
* **算法地位**：如果只看**平均周转时间（ATT）**，在已知任务运行时间的前提下，**STCF 是理论上的最优算法（Provably Optimal）**。

### 4. 轮转法 (Round Robin, RR)
* [cite_start]**核心矛盾**：为了优化**响应时间**，必须打破假设 3（引入时间片概念，允许工作被轮流调度）。
* [cite_start]**调度策略**：引入一个固定的微小时间段，称为**时间片（Time Slice / Scheduling Quantum）**。任务 A 跑完一个时间片，立刻被操作系统移到就绪队列的末尾 ，换任务 B 跑一个时间片 ，大家轮流坐庄。
* **指标权衡**：
  * **响应时间（Response Time）**：**极好**。所有任务都能在极短时间内第一次摸到 CPU。
  * **周转时间（Turnaround Time）**：**极差**。因为大家都慢悠悠地同时向前推进，导致所有任务几乎拖到最后才扎堆完成 。

#### ⚖️ 轮转法的设计艺术：时间片该定多长？
这是系统设计者的一种**权衡（Trade-off）**：
* [cite_start]**时间片太短**：响应时间变好。但**上下文切换（Context-switch）的成本将主导整体性能** 。CPU 绝大部分精力都花在保存和恢复寄存器上，真正干活效率极低 。
* **时间片太长**：上下文切换的成本被有效平摊。但如果长到一定程度，RR 算法就会退化成传统的 FCFS，响应时间变得令人无法忍受。

## 🔄 四、 融合 I/O：走向真实的复杂世界

最后，彻底放宽假设 4：**真实的进程不可能只做 CPU 计算，它们会频繁地发起 I/O 请求。**

* **当进程发起 I/O 请求时**：
  1. 进程状态立刻从运行态被设置为**阻塞态（Blocked）** 。
  2. 调度程序绝不能让 CPU 闲着，应当**立即在 CPU 上调度另一个就绪的进程** 。
* **当 I/O 操作完成时**：
  1. 硬件设备发出一个**硬件中断（Interrupt）**。
  2. 内核将刚才那个等待 I/O 的进程状态由**阻塞态改为就绪态（Ready）**，送回就绪队列 。
  3. 调度器在合适的时机重新调度它运行 。
* **重叠（Overlap）的艺术**：充分利用多道程序设计，让进程 A 在用 CPU 的时候，进程 B 正在读写磁盘。实现 CPU 资源与 I/O 资源在物理时间上的**并行重叠**，让系统整体吞吐量和利用率达到最大化 。

## 🧠 五、 遗留的终极谜题

到目前为止，优秀的调度算法（如 SJF、STCF）能够完美运行，都建立在假设 5 的基础之上：**系统必须预知每个任务要运行多久。**

但在现实的通用操作系统（Linux、Windows、macOS）中，**内核根本不可能预知用户刚刚双击打开的软件会运行几秒钟还是几小时** 。

> ❓ **引出下一章的悬念**：
> 如果我们**对任务的运行时间一无所知**，我们该如何设计调度算法，既能让短任务快速响应，又能让长任务平稳运行？
> 
> 这就是现代操作系统真正采用的终极算法——**多级反馈队列（Multi-Level Feedback Queue, MLFQ）**的由来。
