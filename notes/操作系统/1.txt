# 操作系统1：虚拟化与进程抽象

## 1. 为什么需要进程？（背景与演进）

从单道程序设计向多道程序设计演进的过程中，系统的复杂性催生了进程的概念。

- **单道程序系统**：内存中通常只装入一道用户程序，CPU 仅执行这一道程序。
- **痛点**：由于程序在等待 I/O 等操作时 CPU 只能闲置，导致 **CPU 利用率极低**。
- **多道程序设计（Multiprogramming）**：内存中同时放入多道程序，目的是为了**提升系统的吞吐量**和保持 CPU 繁忙。
- **并发管理带来的新挑战**：
  - 多个程序之间需要频繁切换。
  - 需要精准记录每个程序运行到了哪里（执行状态）。
  - 需要防止多个程序之间互相干扰或影响。

磁盘上“静态”的程序已无法满足多道并发管理的需求，因此操作系统引入了“进程（Process）”这一动态抽象。

---

## 2. 深入理解程序与进程的关系

课件提供了一个非常形象的“厨师做菜”比喻来帮助理解这几个核心操作系统概念：

| 操作系统概念 | 厨房比喻 | 概念解析 |
| --- | --- | --- |
| **程序 (Program)** | **菜谱** | 存在磁盘上的、lifeless（无生命）的指令和数据的集合（如 `.exe` 文件）。 |
| **进程 (Process)** | **做菜的过程** | 程序在计算机中的一次**动态执行过程**，包含状态和资源。 |
| **CPU** | **厨师** | 实际负责执行指令的硬件处理器。 |
| **内存 (Memory)** | **厨房台面** | 存放当前运行所需代码和数据的运行空间。 |
| **数据 (Data)** | **食材** | 进程处理的对象和输入内容。 |

> **核心定义**：进程 = “执行中的程序 + 当前运行状态 + 操作系统资源信息”。

## 3. CPU 虚拟化：时分共享机制

为了让只有有限个 CPU（例如 8 个）的机器产生能够同时运行成百上千个程序的假象（虚拟出很多个 CPU），操作系统采用了 CPU 虚拟化技术。

- **时分共享（Time sharing）**：操作系统让一个进程只运行一个很短的**时间片**，然后迅速切换到其他进程轮流使用 CPU。
- **机制支撑**：依赖上下文切换（Context-switch）机制，它能让操作系统安全地停止一个程序，并在同一个 CPU 上开始运行另一个程序。
- **潜在代价**：会带来一定的**性能（Performance）损失**，因为频繁的切换有开销，且宏观上看每个进程的实际运行速度变慢了。

## 4. 进程的构成与关键数据结构（PCB）

进程在操作系统中不仅是一个运行概念，还对应着一系列复杂的底层数据结构。

### 4.1 进程控制块（PCB, Process Control Block）

PCB 是操作系统为了管理进程而维护的元数据结构，用于表征进程的身份和状态。在 xv6 操作系统中被定义为 `struct proc`：

- **程序计数器（PC, Program Counter）**：记录程序当前正在执行哪一条指令（下一条要执行的指令位置）。
- **寄存器信息（Context）**：在进程被暂停时，CPU 的各个寄存器现场值会被保存在 PCB 的内存中，以便后续重新装入 CPU 恢复运行。
- **栈指针（Stack/Frame pointer）**：管理函数参数栈、局部变量和函数返回地址。
- **I/O 信息（打开文件列表）**：在类 UNIX 系统中，每个进程默认有 3 个打开的文件描述符：
  - `0`：标准输入（`stdin`）
  - `1`：标准输出（`stdout`）
  - `2`：标准错误（`stderr`）
  - 其他数字（如 `3`）表示进程运行时动态打开的其他文件或文件夹。
- **进程标识信息**：如进程 ID（`pid`）、父进程 ID（`ppid`）、用户 ID（`uid`）等。
- **调度与状态信息**：进程的当前状态（如 `proc_state`）和调度优先级。

### 4.2 进程的内存空间组成

当程序载入内存变为进程时，其地址空间主要划分为：

1. **程序代码与静态数据**：从磁盘（ELF/PE 格式）中读取并载入。
2. **栈（Stack）**：用于管理局部变量、函数参数以及函数调用返回地址。
3. **堆（Heap）**：用于程序显式请求的**动态内存分配**（如 C 语言中通过 `malloc()` 申请，通过 `free()` 释放），常用于构建链表、散列表和树等动态数据结构。

## 5. 进程的生命周期与状态转换

进程在其生命周期中会在几种核心状态之间进行转换。

### 5.1 三种基本状态

1. **运行（Running）**：进程正在处理器上实际执行指令。
2. **就绪（Ready）**：进程已准备好运行，但由于 CPU 被其他进程占用，操作系统此时选择不运行它。
3. **阻塞（Blocked）**：进程执行了某种操作（通常是发起了 I/O 请求），正在等待结果返回。此时它会让出 CPU，允许其他进程使用处理器。

### 5.2 状态转换逻辑

- **就绪 -> 运行**：该进程被操作系统**调度（Scheduled）**。
- **运行 -> 就绪**：运行时间片用完，被操作系统**取消调度（Descheduled）**。
- **运行 -> 阻塞**：进程**发起 I/O**（或等待某事件），主动让出 CPU。
- **阻塞 -> 就绪**：**I/O 操作完成**（或等待的事件发生），操作系统将其唤醒放回就绪队列。

此外，还存在进程创建时的**初始状态（Initial）**和退出但尚未被父进程清理的**最终状态/僵尸状态（Final/Zombie）**。

## 6. 进程管理 API 与常用调试工具

### 6.1 核心进程 API

现代操作系统通过系统调用为用户提供管理进程的接口：

- **Create（创建）**：`fork()`、`exec()`、`clone()`。
- **Destroy（销毁）**：`kill(pid, SIGTERM)`。
- **Wait（等待）**：`wait(pid)` 等待子进程停止或结束。
- **Miscellaneous Control（杂项控制）**：如暂停进程 `kill(pid, SIGSTOP)`，恢复运行进程 `kill(pid, SIGCONT)`。
- **Status（状态获取）**：通过读取系统文件（如 `cat /proc/pid/status`）查看进程运行元数据。

### 6.2 实践：Linux 下的进程查看与调试

课件中展示了几个经典的进程诊断与操作指令：

- **查看进程运行时指标**：
  - 使用 `ps aux | grep <进程名>` 或 `htop` 查看进程的 PID、状态（如 `R` 代表运行/就绪，`S` 代表可中断睡眠，`T` 代表调试暂停）。
  - `VmSize` 和 `VmRSS` 分别代表进程占用的虚拟内存大小和实际物理内存（常驻集）大小。
- **GDB 调试器挂载运行进程**：
  - 通过命令 `sudo gdb -p <pid>` 可以直接将调试器附加到正在运行的进程上，以便观察其内部寄存器和内存状态。
  - 调试状态下，进程的状态在系统中会变为 `t (tracing stop)`。
